# 사주 vs 타로 — 운명의 대결

**"동양의 사주와 서양의 타로가 당신의 운세를 놓고 3라운드 배틀을 벌입니다"**

유튜브 [bkamp 채널 라이브](https://www.youtube.com/watch?v=HNjQjW37mpI)에서 교육용으로 시연하며 만들기 시작한 서비스인데, 하다 보니 욕심이 나서 **실제 유료 서비스 급**까지 구현해버렸습니다. 라이브 이후에도 계속 개선해서 완성 후 배포까지 마쳤습니다.

---

## 어떤 서비스?

생년월일 + 성별을 입력하면, **사주(四柱)** 와 **타로**가 각각 당신의 운세를 해석합니다. 7개 주제(연애운, 재물운, 직업운, 건강운, 학업운, 대인관계, 종합운세) 중 궁금한 **3가지를 직접 선택**하면, 그 순서대로 3라운드 배틀이 진행됩니다.

처음에는 연애/재물/종합으로 고정이었는데, 만들다 보니 "왜 내가 궁금한 걸 못 고르지?"라는 생각에 확장했습니다.

---

## 유료 서비스 급이라고 자랑하는 이유

### 사주 엔진

- 천간/지지/십성/지장간/12운성/12신살까지 직접 계산
- 음력↔양력 변환 자체 구현 (외부 API 없이 1900~2050년 커버)
- 지방시 보정(-30분), 궁성/조후/합화 프롬프트 적용
- 신살/길성 ~30종, 격국 판단, 오행 생극 관계도
- 대운/세운 분석 + 연령대별 인생 로드맵

### 타로 엔진

- 78장 풀덱 (메이저 22 + 마이너 56), 정/역방향 해석
- 원소 조화(elemental dignity), 호스슈 7장 스프레드
- 나이/성별 기반 코트 카드 자동 배정, 패턴 분석

만들면서 실제로 이런 프롬프트를 날렸습니다:

> *"대한민국에서 철학원만 50년 동안 운영해 온 사주 명인이 평가해도 자신 있을 정도의 논리 구조를 가지고 있어? 유럽에서 30년 동안 타로점만 봐온 집시 할머니가 평가해도 자신 있을 정도의 논리 구조를 가지고 있어?"*

이 질문 하나로 부족한 부분을 전부 찾아내고 개선했습니다.

## 통계 기능 — 예상외로 제일 재밌는 부분

배틀 결과를 **bkend.ai BaaS**에 저장하고, 다양한 관점에서 사주 vs 타로 선호도를 통계적으로 비교할 수 있습니다:

- **성별**: 남자는 사주파? 여자는 타로파?
- **연령대**: 20대 vs 40대, 누가 사주를 더 신뢰할까?
- **주제별**: 연애운은 타로가 우세? 재물운은 사주가 우세?
- **종합 전적**: 역대 참가자 전체의 사주 vs 타로 승률

---

## bkend.ai — 백엔드 없이 백엔드 기능 구현

결과 공유와 통계 기능을 위해 데이터 저장이 필요했는데, 백엔드 서버를 따로 만들고 싶지 않았습니다. **bkend.ai BaaS**를 붙였더니 REST API 래퍼 하나(~150줄)만으로 전부 해결됐습니다.

```
프론트엔드 → bkend.ai REST API → MongoDB
             (테이블 생성, CRUD, 인덱스, CORS 전부 대시보드에서 클릭 몇 번)
```

별도의 백엔드 코드, ORM, 마이그레이션 스크립트가 하나도 없습니다. API 키는 CORS 도메인 제한으로 보호되니 클라이언트에 노출해도 안전합니다.

### 통계 최적화 — 인덱스 설계와 서버 측 카운팅

처음 AI가 통계를 구현할 때는 전체 데이터를 다 가져와서(~2.5MB) JS로 forEach 집계했습니다. 당연히 제동을 걸었죠:

> *"통계 정보만 받아와야지 왜 결과를 다 받아와? 그리고 1000건 넘으면 어떻게 할건데?"*

개선 결과:

- **경량 통계 테이블 분리** — `battle_results`(공유용, 풀 데이터)와 `battle_stats`(통계용, 카테고리 값만) 분리
- **서버 측 카운팅** — `limit=1` + `andFilters` → `pagination.total`만 사용. 데이터를 한 건도 안 가져오면서 서버가 카운팅만 하는 구조
- **파생값 계산** — `tarot = total - saju`, `female = total - male`로 API 호출 수 절감
- **단일 Promise.all** — 27건 서버 카운팅을 1 network round-trip으로 처리
- **MongoDB 인덱스** — `winner`, `gender`, `birth_year`, 주제별 투표 필드에 인덱스 설정

bkend.ai는 MongoDB 쿼리를 거의 그대로 전달하므로 `andFilters`에 `$gte/$lte` 같은 범위 연산자도 사용 가능합니다. 대시보드에서 클릭 몇 번으로 인덱스를 설정할 수 있어서 사전 준비가 부담 없었습니다.

특히 **출생연도를 저장**하고 연령대는 조회 시 계산하는 설계가 포인트입니다:

> *"age 20대, 30대로 하면 해가 지났을 때 그 사람들이 나이를 먹잖아. 출생 연도를 저장했다가 범위로 셀렉해"*

## 기술 스택 (의외로 심플)

프레임워크 **없이** 순수 HTML/CSS/JS로 만들었습니다.

| 항목 | 스택 |
|------|------|
| 프론트엔드 | Vanilla JS SPA (~5,000줄), IIFE 모듈 패턴 |
| AI 해석 | OpenRouter API + DeepSeek V3.2 |
| 백엔드/DB | bkend.ai BaaS |
| 호스팅 | Cloudflare Pages + Workers (API 키 보호) |
| CI/CD | GitHub Actions |
| 빌드 도구 | **없음** (`npx serve .`이면 끝) |

## 실제로 AI에게 이렇게 시켰습니다 — 프롬프트 사례

이 프로젝트는 **Claude Code + bkit 플러그인**으로 바이브코딩한 과정 자체가 교육 콘텐츠입니다. 실제 세션에서 사용한 프롬프트 중 유의미한 것들을 소개합니다:

### 1. 프로젝트 시작 — 아이디어부터 AI와 함께

```
"오늘은 사주와 타로의 대결 서비스를 만들어 볼거야. 너의 아이디어를 먼저 들려줘"
```

처음부터 구체적 스펙을 주지 않고, AI에게 아이디어를 먼저 제안하게 했습니다.

### 2. 도메인 지식이 없을 때 — AI에게 리드 맡기기

```
"/plan-plus 실제 사주와 타로를 잘 모르는 나도 잘 만들 수 있도록
너가 리드해서 디테일한 정보를 수집하고, 추측하지 말고
실제 사주와 타로 이론을 기반으로 한 서비스를 만들어보자"
```

"추측하지 말고 실제 이론 기반으로"라는 지시가 핵심. 이게 없으면 AI가 대충 만듭니다.

### 3. 품질 기준 제시 — 전문가 관점 요구

```
"대한민국에서 철학원만 50년 운영해 온 사주 명인이 평가해도 자신 있어?
유럽에서 30년 타로점만 봐온 집시 할머니가 평가해도 자신 있어?"
```

모호한 "잘 만들어줘" 대신, 구체적인 전문가 페르소나로 품질 기준을 설정하니 AI가 스스로 부족한 점을 찾아냈습니다.

### 4. 버그 발견 시 — AI에게 직접 테스트 시키기

```
"경고 경고!! 2단계 재물운 단계에서 화면이 안 보이는 버그 발생!
너가 데이터 입력해서 테스트 해보고 버그 수정 해줘"
```

### 5. 비용 고민 — 실용적 질문

```
"우리가 다른 모델을 쓴다면, 어떤 모델이 결과가 준수하면서도
가성비가 좋을까? 서비스를 무료로 오픈하려니 토큰비가 부담돼"
```

실제로 GitHub Models → OpenRouter DeepSeek V3.2로 전환해 비용을 대폭 줄였습니다.

### 6. 사용자 경험 개선 — 구체적 UX 지시

```
"응답이 오래 걸리네. 그냥 '운세 읽는 중...'이라고 하기보다는
사주명인과 타로전문가가 한마디씩 던지는 느낌으로 덜 지루하게.
LLM 호출 없이 미리 100개씩 멘트 뽑아두고 랜덤 사용.
사주명인은 한자어를 섞어가며 전문성 있는 느낌으로,
타로전문가는 신비로운 말투로 기대감을 증폭시키도록"
```

로딩 UX를 구체적으로 설계해서 지시한 사례. "알아서 해줘"보다 훨씬 좋은 결과가 나옵니다.

### 7. 잘못된 구현에 제동 걸기

```
"통계 정보만 받아와야지 왜 결과를 다 받아와?
그리고 1000건 넘으면 어떻게 할건데?"
```

```
"안되겠다. 역시 PDCA plan으로 돌려야해.
현재 통계 정보에 대해 검토하고 개선안을 가져와봐"
```

AI가 비효율적으로 구현했을 때 바로 멈추고, 계획 단계로 되돌린 사례. 바이브코딩에서도 설계가 중요합니다.

### 8. 배포와 보안 — 실무 관점

```
"이 서비스를 실제 배포하고자 하는데, 부족한 점이 있으면 살펴보고,
보안적인 측면도 체크하고, 배포 방법과 비용도 체크해봐.
필요한 적합한 에이전트들 호출해"
```

보안/비용/배포를 한번에 검토. 여러 전문 에이전트를 병렬로 호출해서 분석했습니다.

## 개발 여정 (커밋 30개, 세션 12개)

| 날짜 | 주요 작업 |
|------|-----------|
| 3/5 | 라이브 방송 중 서비스 초안 제작, 기본 배틀 구조 완성 |
| 3/5~7 | 타로 78장 이미지 확보, 사주/타로 엔진 전문가 수준 업그레이드 |
| 3/7~8 | OpenRouter 전환, 로딩 UX 개선, 결과 공유 기능 |
| 3/8~10 | Cloudflare 배포, GitHub Actions CI/CD, 통계 기능 |
| 3/10~11 | 만세력 정확도 교정, 7주제 선택 기능, 인생 로드맵 |
