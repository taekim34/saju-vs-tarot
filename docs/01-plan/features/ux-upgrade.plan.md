# Plan: ux-upgrade (입력 UX + 타로 스프레드 업그레이드)

> Created: 2026-03-07
> Feature: ux-upgrade
> Status: Plan
> Depends on: engine-upgrade (완료)

## 1. Overview

사주 vs 타로 배틀의 **입력 정확도**와 **타로 리딩 깊이**를 한 단계 끌어올리는 UX/엔진 업그레이드.
사주 쪽은 음력/양력 구분을 추가하여 **월주 정확도**를 대폭 향상하고,
타로 쪽은 자유 질문 + 시그니피케이터 + 호스슈 스프레드로 **개인화된 리딩**을 제공한다.

### 업그레이드 요약

| # | 개선항목 | 영향 엔진 | 효과 |
|---|---------|-----------|------|
| U1 | **양력/음력 토글** | saju.js, app.js, index.html | 음력 생일 사용자의 월주 정확도 향상 (전체 사용자의 ~30%) |
| U2 | **자유 질문 입력** | app.js, battle.js, ai-interpreter.js | 사용자 맞춤 해석 → 공감도 향상 |
| U3 | **코트 카드 자동 배정 (시그니피케이터)** | tarot.js | 나이/성별 기반 인물 카드 → 리딩 개인화 |
| U4 | **호스슈 7장 스프레드** | tarot.js | 종합운세 라운드의 해석 깊이 대폭 향상 |
| U5 | **Output 토큰 제한 검토** | config.js, ai-interpreter.js | 응답 품질과 비용의 최적 밸런스 |

## 2. Scope

### In Scope
- 양력/음력 토글 UI + 음력→양력 변환 로직
- 자유 질문 텍스트 입력 필드 + AI 프롬프트 반영
- 나이/성별 → 코트 카드 자동 선택 로직
- R3 종합운세: 쓰리카드 → 호스슈 7장 업그레이드
- MAX_TOKENS 최적화 (응답 길이 vs 비용 밸런스)

### Out of Scope
- 켈틱 크로스(10장) 등 추가 스프레드 (향후 고려)
- 음력 변환을 위한 외부 API 호출 (로컬 테이블 사용)
- AI 모델 변경 (별도 작업)
- 사주 엔진 내부 로직 변경 (engine-upgrade에서 완료)

## 3. Requirements

### 3.1 U1: 양력/음력 토글

**배경**: 한국인의 약 30%가 음력 생일을 사용. 음력 날짜를 양력으로 착각해 입력하면 월주가 완전히 달라져 사주 정확도가 크게 떨어짐.

**구현 내용**:
```
UI 변경:
- 생년월일 입력 그룹 위에 [양력] / [음력] 토글 버튼 추가
- 기본값: 양력 (selected)
- 음력 선택 시 윤달 체크박스 표시

음력→양력 변환:
- data/lunar-calendar.json 생성 (1940~2025년 음력 데이터)
- 음력 날짜 → 양력 날짜 변환 함수 (convertLunarToSolar)
- 윤달 처리 포함

데이터 흐름:
- 사용자가 음력 선택 → 음력 날짜 입력
- battle 시작 시 양력으로 변환 → saju.js에는 항상 양력 전달
- 변환 불가 시 에러 메시지 표시
```

**영향 파일**: `index.html`, `app.js`, `data/lunar-calendar.json` (신규)

**참고**: 음력 변환 데이터는 한국천문연구원 기준. 1940~2025년 범위로 충분 (selectbox 범위와 동일).

### 3.2 U2: 자유 질문 입력

**배경**: 현재 3라운드 주제가 고정(연애운/재물운/종합운세). 사용자가 궁금한 것을 직접 물어볼 수 있으면 공감도가 크게 상승.

**구현 내용**:
```
UI 변경:
- 출생시간 아래에 "궁금한 것이 있나요?" 텍스트 입력 필드 추가
- placeholder: "예: 이직을 고민 중이에요 / 올해 결혼 운은?"
- 선택 사항 (빈칸이면 기존 3라운드 그대로 진행)
- 최대 100자 제한

데이터 흐름:
- 질문이 있으면 → AI 프롬프트의 system/user 메시지에 추가
- 사주: "사용자의 추가 질문: {question}" → 해당 맥락을 운세에 반영
- 타로: "질문자가 특히 궁금해하는 것: {question}" → 카드 해석에 반영
- 3라운드 모두에 질문이 반영됨 (주제별로 다른 각도에서 답변)

주의:
- XSS 방지: 입력값은 반드시 이스케이프 후 프롬프트에 삽입
- 빈 질문이면 기존 로직 그대로 (하위 호환)
```

**영향 파일**: `index.html`, `app.js`, `battle.js`, `ai-interpreter.js`

### 3.3 U3: 코트 카드 자동 배정 (시그니피케이터)

**배경**: 전통 타로에서는 질문자를 대표하는 코트 카드(Page/Knight/Queen/King)를 먼저 선택. 나이/성별로 자동 배정하면 리딩이 더 개인적으로 느껴짐.

**구현 내용**:
```
배정 규칙 (나이 + 성별):
- 10대 이하: Page (남녀 공통)
- 20~30대 남성: Knight
- 20~30대 여성: Queen
- 40대 이상 남성: King
- 40대 이상 여성: Queen

수트 선택 (질문 주제 기반):
- 연애운 → Cups (물/감정)
- 재물운 → Pentacles (흙/물질)
- 종합운세 → 일간(dayMaster) 오행에 맞는 수트
  - 목/화 → Wands
  - 수 → Cups
  - 금/바람 → Swords
  - 토 → Pentacles

시그니피케이터 활용:
- 덱에서 해당 코트 카드를 제거하지 않음 (등장 가능)
- 대신 AI 프롬프트에 "질문자 시그니피케이터: {카드명}" 정보 전달
- 해당 코트 카드가 스프레드에 등장하면 "질문자 자신을 나타냄" 해석 추가
```

**영향 파일**: `tarot.js`, `ai-interpreter.js`

### 3.4 U4: 호스슈 7장 스프레드

**배경**: 현재 R3 종합운세가 5장(파이브카드). 호스슈(Horseshoe) 7장은 종합운세에 최적화된 전통 스프레드로, 더 다양한 각도의 해석 가능.

**구현 내용**:
```
호스슈 스프레드 (7장):
포지션:
  1. 과거 영향 — 현재에 영향을 미치는 과거 사건
  2. 현재 상황 — 지금 처한 상황의 핵심
  3. 숨겨진 영향 — 본인이 인식하지 못하는 요소
  4. 장애물 — 극복해야 할 도전
  5. 주변 환경 — 주변 사람들/상황의 영향
  6. 조언 — 이 시기에 필요한 행동
  7. 최종 결과 — 현재 흐름이 이어질 때의 결과

SPREADS 업데이트:
3: {
  name: '호스슈 스프레드',
  topic: '종합운세',
  count: 7,
  positions: ['과거 영향', '현재 상황', '숨겨진 영향', '장애물', '주변 환경', '조언', '최종 결과']
}

패턴 분석 강화:
- 7장이면 수트 패턴/숫자 패턴이 더 의미있게 작동
- 원소 상성(Elemental Dignity) 분석 쌍이 6개로 증가
- 메이저 아르카나 비율 판단 기준 추가:
  "7장 중 4장 이상 메이저 → 운명이 강력하게 개입하는 시기"
```

**영향 파일**: `tarot.js`

### 3.5 U5: Output 토큰 제한 검토

**배경**: 현재 `Math.max(config.MAX_TOKENS || 0, 4096)` — 최소 4096 토큰. 7장 스프레드로 인해 입력 토큰이 늘어나므로 출력 토큰도 재검토 필요.

**구현 내용**:
```
현재 상태:
- config.js 기본값: MAX_TOKENS = 4096
- ai-interpreter.js: Math.max(config.MAX_TOKENS || 0, 4096) 강제
- GitHub Models gpt-5-mini: 16k context window

분석 필요 항목:
1. 사주 프롬프트 토큰 (시스템 + 유저 메시지) 추정
2. 타로 7장 프롬프트 토큰 추정
3. 이상적 응답 길이 (한글 800~1200자 = 약 1500~2500 토큰)
4. 모델별 토큰 제한 비교:
   - GitHub Models gpt-5-mini: max 16k
   - Gemini 2.0 Flash: max 8k output
   - 미래 모델 이전 고려

조정 방안:
- 사주 리딩: max_completion_tokens = 3000 (충분)
- 타로 리딩 (7장): max_completion_tokens = 4000 (카드 7장 해석)
- 최종 판정: max_completion_tokens = 2000 (간결하게)
→ 라운드별/함수별로 토큰 제한을 다르게 설정하는 구조로 변경

비용 절감 효과:
- 현재: 모든 호출 4096 고정
- 개선: 라운드별 적정 토큰 → 평균 ~3000으로 절감 (~25% 절감)
```

**영향 파일**: `config.js`, `ai-interpreter.js`

## 4. Implementation Order

```

  → 기존 기능에 영향 없이 UI만 추가

  → 사주 엔진에는 항상 양력 날짜만 전달되므로 saju.js 수정 불필요

Phase 3 — 타로 스프레드 + 시그니피케이터 (U3 + U4)
  tarot.js SPREADS[3]: 호스슈 7장으로 변경
  tarot.js: getSignificator(age, gender, topic, dayMasterElement)
  tarot.js: analyzePatterns에 7장 기준 추가
  → 덱 관리는 기존 구조 그대로 (drawCards에서 count만 바뀜)

Phase 4 — AI 프롬프트 통합 + 토큰 최적화 (U2 + U3 + U5)
  ai-interpreter.js: 자유 질문을 프롬프트에 삽입
  ai-interpreter.js: 시그니피케이터 정보 프롬프트에 삽입
  ai-interpreter.js: 함수별 max_completion_tokens 분리
  config.js: 토큰 설정 구조 변경

Phase 5 — 통합 테스트 + 캐시 버스팅
  index.html: 스크립트 버전 업데이트
  전체 3라운드 배틀 테스트
  토큰 사용량/응답 품질 확인
```

## 5. Data Design

### 5.1 음력 데이터 구조 (data/lunar-calendar.json)

```json
{
  "description": "음력→양력 변환 테이블 (1940~2025)",
  "data": {
    "1990": {
      "months": [30, 29, 30, 29, ...],  // 각 음력월의 일수 (29 or 30)
      "leapMonth": 5,                    // 윤달 (없으면 0)
      "leapMonthDays": 29,              // 윤달 일수
      "solarBase": "1990-01-27"         // 음력 1월 1일의 양력 날짜
    },
    ...
  }
}
```

**크기 추정**: ~86개 연도 × ~200B = ~17KB (충분히 가벼움)

### 5.2 시그니피케이터 반환 구조

```javascript
{
  rank: 'queen',          // page/knight/queen/king
  suit: 'cups',           // wands/cups/swords/pentacles
  korean: '컵의 여왕',
  meaning: '내면의 성숙, 감정적 지혜를 가진 질문자',
  reason: '30대 여성 + 연애운 주제'
}
```

### 5.3 battle.js userData 확장

```javascript
// 기존
{ year, month, day, gender, hour, minute }

// 확장
{ year, month, day, gender, hour, minute,
  calendarType: 'solar' | 'lunar',  // U1
  isLeapMonth: false,                // U1
  question: '이직 고민...'           // U2
}
```

## 6. Success Criteria

| 항목 | 기준 |
|------|------|
| 양력/음력 토글 | 음력 입력 → 양력 변환 정확 (1990년 음력 8/15 = 양력 10/3 확인) |
| 자유 질문 | 질문이 AI 해석에 반영됨 (빈 질문이면 기존과 동일) |
| 시그니피케이터 | 나이/성별/주제에 맞는 코트 카드 자동 선택 확인 |
| 호스슈 스프레드 | R3에서 7장 정상 드로우 + 포지션명 정확 + 패턴 분석 작동 |
| 토큰 최적화 | 함수별 토큰 제한 분리, 평균 ~25% 비용 절감 |
| 하위 호환 | 양력 선택(기본값) + 빈 질문이면 기존과 100% 동일 동작 |
| 전체 배틀 | 3라운드 정상 완료, R3의 7장 해석이 기존 5장보다 풍부 |

## 7. Risk

| 리스크 | 대응 |
|--------|------|
| 음력 데이터 파일이 너무 클 수 있음 | 86년분 ~17KB로 가벼움, gzip 시 ~5KB |
| 음력 변환 라이브러리 버그 | 검증된 공개 만세력 데이터 사용, 10개 이상 날짜 교차 검증 |
| 7장 스프레드로 AI 응답이 너무 길어짐 | max_completion_tokens를 4000으로 제한, 프롬프트에 "핵심만 간결하게" 지시 |
| 7장이면 덱 소진이 빨라짐 | R1(3) + R2(3) + R3(7) = 13장, 78장 덱에서 충분 |
| 자유 질문에 XSS/프롬프트 인젝션 | escapeHtml + 100자 제한 + AI에 "사용자 질문은 운세 맥락으로만 해석" 지시 |
| 시그니피케이터가 스프레드에 중복 등장 | 의도적으로 허용 — "질문자 자신이 직접 등장" 해석으로 활용 |
| 토큰 비용이 예상보다 높을 수 있음 | 7장 → 사주(3000)+타로(4000) = 한 라운드 7000 토큰, 전체 ~21K → 모델별 비용 재계산 필요 |
